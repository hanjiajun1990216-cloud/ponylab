# 第一部分：产品愿景与战略

## 第 1 章：执行摘要

### 1.1 产品一句话定位

**Ponylab 是一款 AI 原生的开源实验室信息管理系统（LIMS + ELN），为 10-500 人规模的研究实验室和检测机构提供从实验设计到合规报告的全链路数字化管理平台。**

### 1.2 核心价值主张

Ponylab 解决的核心问题：**科研数据的碎片化管理导致知识流失、合规风险和效率低下。**

当前全球科研实验室面临三大结构性痛点：

1. **数据孤岛**：实验记录在纸质笔记本、Excel、Word、仪器电脑之间碎片化分布，每年约 30% 的实验室知识随人员流动而永久丢失
2. **合规负担**：FDA 21 CFR Part 11、GLP/GMP、ISO 17025 等合规要求日益严格，手动维护审计追踪耗费大量时间
3. **效率瓶颈**：仪器数据手动转录、库存人工盘点、报告逐项编写——这些可自动化的工作占据了研究人员 20-40% 的时间

**Ponylab 的差异化解决方案**：

| 竞品痛点                             | Ponylab 方案                                  |
| ------------------------------------ | --------------------------------------------- |
| Benchling 价格极高（$5K-7K/用户/年） | 开源免费 + 商业版透明定价（$15/用户/月起）    |
| SciNote 表格体验差、AI 能力弱        | 类 Excel 表格编辑 + AI 原生集成               |
| LabWare/STARLIMS UI 陈旧、部署耗时   | 现代 UI + 2 周内可上线                        |
| 所有竞品 AI 停留在搜索层面           | RAG 知识库 + 自然语言查询 + 异常检测          |
| 大多数产品不支持中国市场             | 企业微信/钉钉集成 + 国密算法 + 完全私有化部署 |

**可服务市场（SAM）**：10-500 人规模的生命科学、化学、材料科学、环境检测实验室
**可获得市场（SOM）**：首年目标 1000 个免费用户 + 50 个付费团队

## 第 2 章：市场分析

### 2.1 全球 LIMS/ELN 市场格局

#### 2.1.1 市场规模与增长

全球 LIMS 市场在 2024 年达到 $24.4-28.8 亿美元，预计到 2030 年将增长至 $35.6-51.9 亿美元，复合年增长率（CAGR）在 6.2%-12.5% 之间。

**核心增长驱动力**：

1. **云化加速**：SaaS 模式占据最大份额且增长最快
2. **AI 整合**：主流厂商纷纷宣称 AI 能力，但真实深度参差不齐
3. **合规强化**：FDA、EMA 对电子记录要求趋严
4. **数字化转型**：COVID-19 后实验室数字化需求激增
5. **开放科学运动**：数据共享和可重复性要求推动标准化

#### 2.1.2 市场特征

- **高度碎片化**：429+ 活跃竞争者，无单一主导者
- **三层竞争格局**：

| 层级       | 代表产品                           | 特征                                    | 年费区间    |
| ---------- | ---------------------------------- | --------------------------------------- | ----------- |
| 企业重量级 | LabWare, STARLIMS, LabVantage      | 功能全面、合规强、UI 老旧、实施 9-18 月 | $50K-$500K+ |
| 现代化中端 | Benchling, Sapio, Labguru, SciNote | 体验好、特定领域强、各有短板            | $5K-$200K   |
| 轻量/垂直  | CloudLIMS, Quartzy, OpenSpecimen   | 特定场景强、通用能力弱                  | $900-$10K   |

## 第 4 章：产品定位与差异化

### 4.1 核心差异化策略

Ponylab 的差异化基于 **"中端三角"** 定位：

```
                    现代化 UI/UX
                        △
                       / \
                      /   \
                     / ★   \    ← Ponylab 占据中心
                    /Ponylab\
                   /─────────\
      完整 LIMS 功能 ───────── 合规能力
```

**没有任何现有产品同时满足这三个维度**：

- Benchling：UI 好 + 功能强 → 但价格极高、合规弱
- SciNote：开源 + 合规 → 但 UI 差、功能有限
- STARLIMS：功能全 + 合规强 → 但 UI 极旧、部署极慢
- eLabFTW：开源 + 好用 → 但缺 LIMS 功能、无合规

### 4.2 产品哲学

**三个核心原则**：

#### 原则 1: AI 原生（AI-Native）

不是在传统 LIMS 上"加装"AI，而是从数据模型层面为 AI 设计。

- 所有实验数据都有结构化标注，便于 AI 消费
- 自然语言查询是一等公民，不是附加功能
- AI 建议嵌入工作流（如：自动推荐相关协议、检测库存不足、预测实验结果）

#### 原则 2: 开放生态（Open Ecosystem）

- 核心开源（AGPL-3.0 或 BSL-1.1）
- 完整 REST + GraphQL API
- 数据可导出（CSV、JSON、SISA 格式）
- 无供应商锁定

#### 原则 3: 渐进式复杂度（Progressive Complexity）

- 新用户 5 分钟内完成第一个实验记录
- 高级功能按需激活（不用合规时不显示合规选项）
- 配置优于定制（低代码配置实验室工作流）

#### 6.1.1 数据模型

```

        └── Task（任务/步骤）
              ├── metadata: 名称、描述、状态、指派人、截止时间
              ├── Protocol（关联实验方案）
              ├── Notes（富文本笔记，支持版本历史）
              ├── Results（实验结果）
              │     ├── Tables（数据表格）
              │     ├── Files（文件附件：图片、PDF、仪器数据）
```

**技术选型**：TipTap（基于 ProseMirror）

#### 6.1.3 实验状态工作流

```
Draft（草稿）
  ↓ [提交审核]
In Review（审核中）
  ↓ [审核通过] / ← [驳回修改]
Approved（已批准）
  ↓ [归档]
Archived（已归档）
```

**状态变更规则**：

- Draft → In Review：需填写必填字段（标题、实验目的）
- In Review → Approved：需指定审批人签名（合规模式下需电子签名）
- Approved → Archived：自动归档（保留不可修改）
- 任何状态变更自动记录审计日志

#### 6.1.4 搜索功能

- **全文搜索**：搜索实验记录、笔记、附件内容（PDF OCR）
- **结构化搜索**：按项目、日期、状态、标签、成员筛选
- **AI 语义搜索**：自然语言查询（如"去年做过的 PCR 实验"）
- **实时搜索建议**：输入时自动补全
- **搜索引擎**：Meilisearch（中文支持好、运维简单）

### 6.8 合规与审计

#### 6.8.1 审计追踪设计

遵循 **ALCOA+ 原则**（Attributable, Legible, Contemporaneous, Original, Accurate + Complete, Consistent, Enduring, Available）：

```
AuditLog（审计日志 — 只追加，禁止 UPDATE/DELETE）
  ├── id: BIGINT AUTO_INCREMENT
  ├── timestamp: 精确到毫秒
  ├── user_id: 操作人
  ├── user_ip: 操作 IP
  ├── action: 操作类型（CREATE / UPDATE / DELETE / VIEW / SIGN / LOGIN / EXPORT）
  ├── entity_type: 实体类型（Experiment / Sample / InventoryItem / Protocol...）
  ├── entity_id: 实体 ID
  ├── field_name: 变更字段名
  ├── old_value: 变更前值（加密存储）
  ├── new_value: 变更后值（加密存储）
  ├── metadata: 额外元数据（JSONB）
  └── hash: 行级哈希（用于完整性验证）
```

**关键设计决策**：

- 审计日志表**只允许 INSERT**，数据库层面禁止 UPDATE 和 DELETE
- 每行包含 SHA-256 哈希，链式验证数据完整性
- 事件驱动审计（持久化后发布事件，审计逻辑解耦）
- GDPR 删除权通过**假名化**实现（替换用户标识），不物理删除审计记录

#### 6.8.2 电子签名

```
ElectronicSignature（电子签名）
  ├── id: UUID
  ├── signer_id: 签名人
  ├── entity_type / entity_id: 签名对象
  ├── action: 签名动作（Approve / Review / Witness / Author）
  ├── meaning: 签名含义（"I have reviewed and approve"）
  ├── signature_method: 签名方式（Password / MFA / Certificate）
  ├── timestamp: 签名时间
  ├── ip_address: 签名 IP
  └── hash: 签名哈希（签名对象 + 时间 + 用户 的哈希）
```

**21 CFR Part 11 合规要求实现**：

| 要求         | 实现方式                           |
| ------------ | ---------------------------------- |
| 唯一用户标识 | 用户名 + UUID，不可重复使用        |
| 电子签名绑定 | 签名 = 密码再次验证 + 签名含义声明 |
| 审计追踪     | 只追加日志，行级哈希，不可篡改     |
| 权限控制     | RBAC + ABAC，最小权限原则          |
| 会话管理     | 自动超时、单点登录限制             |
| 数据备份     | 每日自动备份，异地容灾             |
| 系统验证     | IQ/OQ/PQ 验证文档                  |

#### 6.8.3 权限模型

采用 eLabFTW 验证的 **五层权限模型**，并增加 ABAC 策略：

```
层级 1: 系统角色（System Role）
  └── Super Admin / Org Admin / User

层级 2: 团队角色（Team Role）
  └── Team Owner / Team Admin / Team Member / Team Viewer

层级 3: 实体所有权（Ownership）
  └── Creator 拥有完全控制权

层级 4: 团队组（Team Group）
  └── 团队内子分组的权限继承

层级 5: 个人授权（Individual Permission）
  └── 对特定用户的特定资源授予/撤销权限

ABAC 策略层:
  └── 基于属性的访问控制（如：合规实验必须双人签名）
```

### 13.4 多租户策略

```
普通客户 → PostgreSQL Row Level Security (RLS)
  - 所有表增加 tenant_id 列
  - RLS 策略自动过滤数据
  - 共享数据库，成本低

制药/GxP 客户 → 独立数据库实例
  - Kubernetes Namespace 隔离
  - 独立 PostgreSQL 实例
  - 完全数据隔离
  - 成本较高
```

### 15.4 安全措施

| 措施         | 实现方式                                   |
| ------------ | ------------------------------------------ |

| 依赖安全     | Dependabot + npm audit                     |
| 访问日志     | 完整 HTTP 日志 + 审计追踪                  |

## 附录 C：开源许可证风险评估

| 许可证      | 风险等级 | 说明                                                          |
| ----------- | -------- | ------------------------------------------------------------- |
| AGPL-3.0    | 高       | SaaS 提供服务需开源全部代码。可学习设计思路，**不要复制代码** |
| GPL-2.0/3.0 | 高       | 传染性强，商业使用风险高                                      |
| MPL-2.0     | 中       | 修改 MPL 文件需开源，整体项目可闭源。**可参考代码**           |
| BSD-3       | 低       | 商业友好。**可参考代码**（如 OpenSpecimen）                   |

**Ponylab 推荐许可证**：BSL-1.1（Business Source License）或 AGPL-3.0

- BSL-1.1：源码公开但商业使用需授权，N 年后自动转为开源
- AGPL-3.0：真开源但 SaaS 竞争者也需开源
