# 第一部分：产品愿景与战略

## 1.1 产品定位

**一句话定位**：PonyLab 是面向中小型科研团队的一站式智能实验室管理平台（ELN + LIMS + AI），致力于成为「科研团队的 Notion + GitHub」。

**核心价值主张**：

- 🧪 **一站式管理**：实验记录、样品、库存、仪器、协议、项目全链路覆盖
- 🤖 **AI 原生**：内置 AI 助手，自动化繁琐操作（语音转录、协议解析、报告生成）
- 👥 **协作优先**：团队机制、留言板、@提及、实时通知，科研不再是孤岛
- 💰 **开源+SaaS**：核心开源，高级功能订阅制，对标 eLabFTW 的开放性 + SciNote 的完整度

```
         │                                     │
```

**PonyLab 的蓝海机会**：

1. **仪器预约日历**：Benchling/SciNote/eLabFTW 均无原生支持，仅 Labguru 有
2. **AI 原生集成**：eLabFTW/LabArchives 无 AI；SciNote 仅有 PDF 导入
3. **项目多层级看板**：竞品中仅 SciNote 有画布，但无 LabManager 式的「方向→项目→任务→步骤」四层结构
4. **开源+AI+仪器预约**三合一：目前市场上没有同时满足这三项的产品

# 第二部分：权限系统与团队机制

## 2.1 权限模型总体架构

### 设计原则

借鉴 SciNote 三层叠加 + Benchling 最高权限胜出 + eLabFTW 条目级权限，设计 PonyLab 的**四层权限继承模型**：

```
层级 1：系统级（System）
    ↓ 默认继承
层级 2：团队级（Team）
    ↓ 默认继承，可覆盖
层级 3：项目级（Project）
    ↓ 默认继承，可覆盖
层级 4：资源级（Resource）— 实验/任务/仪器/库存
```

### 继承规则

1. **向下继承**：高层角色自动赋予低层相同或更低权限
2. **局部覆盖**：低层可对特定用户设置更严格（但不能更宽泛）的权限
3. **最高权限胜出**：用户同时属于多个团队时，取最宽松权限（防权限死锁）
4. **功能角色叠加**：功能型角色（仪器管理员、仓库管理员）与层级角色取并集

## 2.2 角色体系

### 系统角色（3 种）

| 角色           | 英文标识    | 权限范围                                         |
| -------------- | ----------- | ------------------------------------------------ |
| **超级管理员** | SUPER_ADMIN | 全系统管控：用户管理、全局设置、所有团队隐式只读 |
| **系统管理员** | ADMIN       | 用户管理、系统设置，不能直接修改业务数据         |
| **普通用户**   | USER        | 基础使用权限，依赖团队角色获取具体操作权         |

## 2.4 团队机制

### 2.4.1 团队创建

**流程**：

```
用户点击「创建团队」→ 填写团队信息（名称、描述、头像、可见性）
→ 选择可见性模式 → 创建成功，创建者自动成为 TEAM_OWNER
```

**团队可见性模式**（借鉴 Notion Teamspace）：

| 模式        | 描述                         | 适用场景               |
| ----------- | ---------------------------- | ---------------------- |
| **Open**    | 所有系统用户可查看和申请加入 | 公共仪器中心、核心设施 |
| **Closed**  | 可见但需审批才能加入         | 研究组、课题组         |
| **Private** | 不可见，仅通过邀请加入       | 保密项目、企业团队     |

### 2.4.2 邀请机制

**三轨邀请**（借鉴 GitLab + Notion）：

| 方式         | 场景           | 实现细节                                                                                          |
| ------------ | -------------- | ------------------------------------------------------------------------------------------------- |
| **邮件邀请** | 精确邀请特定人 | 发送含一次性 Token 的邮件链接；可预设角色；支持多邮箱批量邀请；2/5/10 天未响应自动提醒；90 天过期 |
| **邀请链接** | 团队内快速拉人 | 生成带有效期的链接（1天/7天/30天/永久）；可限制最大使用次数；可预设默认角色（默认 MEMBER）        |
| **邀请码**   | 线下场景       | 6 位大写字母+数字码；可限制使用次数；通过系统内输入框使用                                         |

### 2.4.3 申请加入

**流程**：

```
用户发现 Open/Closed 团队 → 点击「申请加入」→ 填写申请理由
→ TEAM_OWNER / TEAM_ADMIN 收到通知 → 审批（通过/拒绝+理由）
→ 通知申请人结果
```

**审批规则**：

- 48 小时未处理 → 自动提醒审批人
- 7 天未处理 → 升级提醒至 TEAM_OWNER
- 可设置自动通过规则（如：同机构邮箱后缀自动通过）

### 2.4.4 团队资源共享模型

| 资源类型 | 共享范围             | 说明                                            |
| -------- | -------------------- | ----------------------------------------------- |
| 实验记录 | 团队内               | 默认团队内可见，可单独设为 Private              |
| 项目     | 团队内               | 属于团队，团队成员可查看                        |
| 仪器     | **全局**             | 仪器属于系统级资源，所有团队均可预约            |
| 库存     | 团队内，可跨团队共享 | 默认团队内，INVENTORY_ADMIN 可开放给其他团队    |
| 协议/SOP | 双层仓库             | 个人协议 + 团队协议库（参考 SciNote 设计）      |
| 公告     | 按范围               | 仪器公告=全局；库存公告=团队内；团队公告=团队内 |

## 2.5 数据模型变更

### 新增模型

```prisma
// ─── 团队邀请 ───
model TeamInvitation {
  id           String           @id @default(cuid())
  teamId       String           @map("team_id")
  inviterId    String           @map("inviter_id")
  type         InvitationType   // EMAIL, LINK, CODE
  email        String?          // 邮件邀请时的目标邮箱
  token        String           @unique // 邀请令牌
  code         String?          @unique // 邀请码（6位）
  role         TeamMemberRole   @default(MEMBER)
  status       InvitationStatus @default(PENDING)
  maxUses      Int?             @map("max_uses")
  usedCount    Int              @default(0) @map("used_count")
  expiresAt    DateTime         @map("expires_at")
  createdAt    DateTime         @default(now()) @map("created_at")

  team         Team             @relation(fields: [teamId], references: [id])
  inviter      User             @relation(fields: [inviterId], references: [id])

  @@map("team_invitations")
}

enum InvitationType { EMAIL LINK CODE }
enum InvitationStatus { PENDING ACCEPTED EXPIRED REVOKED }

// ─── 加入申请 ───
model TeamApplication {
  id           String              @id @default(cuid())
  teamId       String              @map("team_id")
  userId       String              @map("user_id")
  reason       String?
  status       ApplicationStatus   @default(PENDING)
  reviewerId   String?             @map("reviewer_id")
  reviewNote   String?             @map("review_note")
  createdAt    DateTime            @default(now()) @map("created_at")
  reviewedAt   DateTime?           @map("reviewed_at")

  team         Team                @relation(fields: [teamId], references: [id])
  user         User                @relation("applicant", fields: [userId], references: [id])
  reviewer     User?               @relation("reviewer", fields: [reviewerId], references: [id])

  @@unique([teamId, userId])
  @@map("team_applications")
}

enum ApplicationStatus { PENDING APPROVED REJECTED }

// ─── 功能角色分配 ───
model FunctionalRole {
  id           String             @id @default(cuid())
  userId       String             @map("user_id")
  teamId       String?            @map("team_id")
  role         FunctionalRoleType
  assignedBy   String             @map("assigned_by")
  createdAt    DateTime           @default(now()) @map("created_at")

  user         User               @relation(fields: [userId], references: [id])
  team         Team?              @relation(fields: [teamId], references: [id])

  @@unique([userId, teamId, role])
  @@map("functional_roles")
}

enum FunctionalRoleType { INSTRUMENT_ADMIN INVENTORY_ADMIN ELN_WITNESS }
```

# 第三部分：项目管理系统

## 3.1 多层级结构设计

借鉴 LabManager 的「方向→项目→任务→步骤」四层结构 + SciNote 的画布 Canvas 设计，PonyLab V2 采用以下层级：

```
研究方向 (Direction)
    └── 实验项目 (Project)
            └── 研究任务 (Task)
                    └── 执行步骤 (TaskStep)
```

### 3.1.1 研究方向 (Direction)

**定位**：最高级别的组织容器，代表一个大的研究主题或科研方向。

| 字段        | 类型            | 说明                                   |
| ----------- | --------------- | -------------------------------------- |
| id          | cuid            | 主键                                   |
| name        | String          | 方向名称                               |
| description | String?         | 方向描述                               |
| leadId      | String          | 方向负责人（FK → User）                |
| teamId      | String          | 所属团队（FK → Team）                  |
| status      | DirectionStatus | ACTIVE / PAUSED / COMPLETED / ARCHIVED |
| color       | String?         | 标识色（用于看板）                     |
| sortOrder   | Int             | 排序序号                               |
| createdAt   | DateTime        | 创建时间                               |

**页面设计**：

```
团队主页 → 点击「研究方向」→ 进入方向列表页
┌─────────────────────────────────────────────────────────┐
```


### 3.1.3 研究任务 (Task) — 画布/看板视图

**定位**：项目下的具体研究任务。这是项目管理的核心交互单元。

### 3.1.4 执行步骤 (TaskStep)

**定位**：任务的细粒度执行步骤，恢复 LabManager Gemini 版的 Step 层。

```prisma
model TaskStep {
  id          String         @id @default(cuid())
  taskId      String         @map("task_id")
  title       String
  description String?
  status      StepStatus     @default(PENDING)
  sortOrder   Int            @default(0)
  assigneeId  String?        @map("assignee_id")
  dueDate     DateTime?      @map("due_date")
  completedAt DateTime?      @map("completed_at")
  completedBy String?        @map("completed_by")
  notes       String?
  createdAt   DateTime       @default(now()) @map("created_at")

  task        Task           @relation(fields: [taskId], references: [id], onDelete: Cascade)
  assignee    User?          @relation(fields: [assigneeId], references: [id])

  @@index([taskId])
  @@map("task_steps")
}

enum StepStatus { PENDING IN_PROGRESS COMPLETED SKIPPED }
```

# 第四部分：仪器管理系统

## 4.1 仪器预约日历

### 设计定位

这是 PonyLab 的**核心差异化功能**。Benchling、SciNote、eLabFTW 均无原生仪器预约日历，仅 Labguru 有完善实现。PonyLab 将提供市场最佳的仪器预约体验。

### 4.1.1 日历视图设计

**资源视图（主视图）**——横轴时间，纵轴仪器：

```
┌─────────────────────────────────────────────────────────────┐
│  📅 仪器预约日历     [日] [周] [月]   ← 2026-03-05 →      │
├─────────────────────────────────────────────────────────────┤
│          │ 08:00  │ 10:00  │ 12:00  │ 14:00  │ 16:00  │   │
│ ─────────┼────────┼────────┼────────┼────────┼────────┤   │
│ UV-Vis   │████████│        │████████│████████│        │   │
│ 分光光度计│ 张三   │        │ 李四   │ 李四   │        │   │
│ ─────────┼────────┼────────┼────────┼────────┼────────┤   │
│ HPLC     │        │████████│████████│        │████████│   │
```

**核心要求**：

- **每个时间块显示预约人姓名**（用户明确要求）

- **维护时段标识**：用特殊图案标记维护窗口期
- **悬浮详情**：鼠标悬浮显示完整信息（预约人、用途、联系方式、备注）

### 4.1.2 日历视图类型

| 视图           | 说明                                 | 适用场景     |
| -------------- | ------------------------------------ | ------------ |

| **周视图**     | 以天为单位，显示一周占用（默认视图） | 本周规划     |
| **月视图**     | 以天为单位，显示一月概览             | 长期规划     |
| **单仪器视图** | 只看一台仪器的日历                   | 预约特定仪器 |

### 4.1.3 预约流程

```
选择仪器 → 选择时间段 → 即时冲突检测
   │                         │
   │  ✅ 无冲突               │  ❌ 冲突
   │                         │
   ↓                         ↓
填写预约信息              提示「已被 [李明] 预约，
（用途、备注）            建议最近空档：14:00-16:00」
   │
   ↓
提交预约 → [需审批仪器] → INSTRUMENT_ADMIN 审批 → 通知结果
           [无需审批]  → 直接确认 → 自动加入日历
```

**冲突检测**（后端已有，需增强前端交互）：

```typescript
{
  "available": false,
  "conflictWith": {
    "userId": "xxx",
    "userName": "李明",

  },
  "nextAvailable": {
    "startTime": "2026-03-05T14:00:00",

  }
}
```

## 5.2 库存增强功能

### 5.2.1 低库存预警

```
当 item.quantity < item.minQuantity 时：

→ 通知 INVENTORY_ADMIN + TEAM_OWNER
→ Dashboard 低库存预警区块显示
```

### 5.2.2 库存公告

参考仪器公告机制，复用 Announcement 模型（scope = INVENTORY）。

**展示位置**：库存列表页顶部横幅，用于通知库存相关公告（如：新试剂到货、某试剂召回、供应商变更等）。

### 5.2.3 自定义列（借鉴 SciNote）

库存表格支持用户自定义列类型，覆盖各种物料：

```prisma
model InventoryColumn {
  id              String           @id @default(cuid())
  teamId          String           @map("team_id")
  name            String
  type            ColumnType       // TEXT, NUMBER, DATE, DROPDOWN, FILE, BARCODE, URL
  options         Json?            // DROPDOWN 时的选项列表
  isRequired      Boolean          @default(false)
  sortOrder       Int              @default(0)
  createdAt       DateTime         @default(now()) @map("created_at")

  team            Team             @relation(fields: [teamId], references: [id])

  @@map("inventory_columns")
}

enum ColumnType { TEXT NUMBER DATE DROPDOWN FILE BARCODE URL }
```

### 5.2.4 条形码/二维码

- 每个库存条目和样品自动生成唯一 QR 码
- 编码格式：`PL-{itemId}` 或 `PL-S-{sampleId}`
- 支持 PDF 标签导出（兼容 Zebra 标签打印机）
- 前端扫码组件（调用摄像头）用于快速定位库存条目

### 5.2.5 库存与任务关联

任务执行时引用库存试剂 → 自动扣减库存数量（借鉴 SciNote 设计）：

```prisma

  taskId          String         @map("task_id")
  inventoryItemId String         @map("inventory_item_id")
```

# 第六部分：实验记录 ELN 核心

## 6.1 实验签署流程增强

### 状态机（借鉴 SciNote + Labii）

```
DRAFT → IN_PROGRESS → COMPLETED → SUBMITTED → SIGNED → WITNESSED → ARCHIVED
                          ↑                      │         │
                          └──── REJECTED ←───────┘         │
                          ↑                                │
                          └──── REJECTED ←─────────────────┘
```

| 状态        | 说明                       | 操作权限          |
| ----------- | -------------------------- | ----------------- |
| DRAFT       | 草稿，可自由编辑           | 作者              |
| IN_PROGRESS | 实验进行中                 | 作者              |
| COMPLETED   | 实验完成，等待提交审签     | 作者              |
| SUBMITTED   | 已提交审签请求             | 作者提交          |
| SIGNED      | PI/PROJECT_LEAD 已签署     | 签署人签署        |
| WITNESSED   | ELN_WITNESS 已见证（可选） | 见证人见证        |
| REJECTED    | 审签被拒绝，退回修改       | 签署人/见证人拒绝 |
| ARCHIVED    | 归档（不可修改、不可删除） | TEAM_OWNER        |

**关键规则**：

1. 已签署记录**不可编辑、不可删除**（21 CFR Part 11）
2. 拒绝后退回 DRAFT，之前的签署失效，需重新走流程
3. 所有状态变更写入审计追踪

## 11.2 RBAC 中间件实现

### NestJS Guard 增强

当前 `RolesGuard` 仅检查系统角色。V2 需要支持多层权限检查：

```typescript
// 新版 PermissionGuard 伪代码
@Injectable()
export class PermissionGuard implements CanActivate {
  async canActivate(context: ExecutionContext): boolean {
    const requiredPermissions = this.reflector.get<Permission[]>(
      "permissions",
      context.getHandler(),
    );
    if (!requiredPermissions) return true;

    const user = context.switchToHttp().getRequest().user;
    const params = context.switchToHttp().getRequest().params;

    for (const perm of requiredPermissions) {
      // 1. 检查系统角色
      if (perm.systemRoles && perm.systemRoles.includes(user.role)) continue;

      // 2. 检查团队角色
      if (perm.teamRoles && params.teamId) {
        const teamMember = await this.getTeamMember(user.id, params.teamId);
        if (teamMember && perm.teamRoles.includes(teamMember.role)) continue;
      }

      // 3. 检查功能角色
      if (perm.functionalRoles) {
        const funcRoles = await this.getFunctionalRoles(user.id, params.teamId);
        if (funcRoles.some((r) => perm.functionalRoles.includes(r))) continue;
      }

      // 4. 检查资源级权限（如：是否是实验作者）
      if (perm.resourceCheck) {
        if (await perm.resourceCheck(user, params)) continue;
      }

      throw new ForbiddenException("Insufficient permissions");
    }
    return true;
  }
}
```
